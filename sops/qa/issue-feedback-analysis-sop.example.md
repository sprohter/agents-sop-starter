## 1. 适用场景

适用于用户、业务、客服、测试或运营反馈的问题分析，包括功能异常、配置不生效、数据差异、外部依赖异常和待确认缺陷。

## 2. 不适用场景

- 已明确只是日常功能测试执行，不需要做反馈分诊
- 已经进入专项性能、容量或基础设施事故处置
- 缺少最小对象和时间范围，且当前无法补齐必要上下文

## 3. 输入

执行前需要：

- 反馈原文、截图、录屏或错误提示
- 时间范围、影响对象、环境信息
- 相关操作入口、样本对象或最小复现路径

## 4. 流程

1. 先复述问题现象，确认当前分析对象、时间范围和影响范围一致。
2. 做第一轮分诊，区分功能问题、数据问题、配置问题、外部依赖问题或环境阻断。
3. 优先收集直接证据：页面现象、错误提示、关键日志、接口返回、状态记录或数据对比。
4. 建立最小事实链，先回答“发生了什么”，再解释“为什么会这样”。
5. 根据证据判断当前应在哪个分支闭环：业务解释、转外部跟进、转需求评估或进入缺陷修复链路。
6. 需要研发介入时，输出最小可复现问题描述、影响范围和关键证据。
7. 收口分析结果，明确当前已确认结论、待补证据、风险和下一步责任归属。

## 5. 判断点

| 判断点 | 处理方式 |
|--------|----------|
| 反馈描述与实际证据不一致 | 先说明不一致点，再继续补证据 |
| 证据只说明现象，尚不能说明根因 | 结论只写已确认现象，根因标待确认 |
| 已可判断为使用方式或口径问题 | 直接给解释和建议，不强行提缺陷 |
| 已确认需要研发修复 | 输出缺陷描述并进入修复跟踪 |

## 6. 输出

默认输出：

- 当前结论
- 已执行步骤
- 关键依据
- 风险或未确认项
- 下一步动作

## 7. 验收标准

满足以下条件才算完成：

- 已明确反馈对象、影响范围和问题分类
- 结论能区分事实、推测和待确认项
- 已明确当前闭环方式或下一步责任归属
