# 自动纠偏机制

> 当 agent 发现自己走错路、证据不足、输出偏移或触碰安全边界时，先纠偏，再继续执行。

## 1. 什么时候触发纠偏

| 触发场景 | 立即动作 |
|----------|----------|
| 结论缺少证据，或使用“应该 / 一般 / 可能”当事实 | 标注为推测，补查文件、记录、日志、配置或用户输入 |
| 读错入口、路由不准、命中了不相关 SOP | 停止当前路径，回到 `routing.md` 和 `routing-maintenance.md` 重新分流 |
| 输出对象看不懂、重点靠后、过程淹没结论 | 重写为“结论 / 依据 / 风险 / 下一步”，必要时补表格 |
| 新增内容不知道放哪里 | 回到 `maintenance-rules.md` 和 `sedimentation.md` 判断层级 |
| 发现敏感信息、真实链接、凭证或运行态噪音 | 停止外发，按 `privacy-and-share-boundary.md` 脱敏或移除 |
| 准备 push、删除、发布、改自动化或写外部系统 | 回到 `change-control.md`，说明目标、影响、回退和验证 |
| 定时任务可能新开会话、弹窗口或无状态文件 | 回到 `automation-rules.md`，改为续接线程、静默运行、写日志和状态 |
| 同一类错误第二次出现 | 不只修当前结果，还要判断是否补路由、SOP、knowledge 或 governance |

## 2. 纠偏步骤

1. **停下当前错误路径**：不要在错误前提上继续扩写。
2. **说清偏差类型**：是证据不足、路由错误、表达问题、安全问题，还是流程门禁缺失。
3. **回到真源**：优先读取 `routing.md`、相关 SOP、knowledge 或 governance。
4. **修正当前交付**：让本次输出先回到可用状态。
5. **判断是否沉淀**：如果下次还会复发，补 `routing.md`、SOP、knowledge 或治理规则。
6. **验证修正结果**：用回读、diff、清单或最小测试证明偏差已消除。

## 3. 常见纠偏样式

| 偏差 | 推荐修正 |
|------|----------|
| 把内部细节写进 public 文档 | 删除细节，改为占位符或抽象示例 |
| 把一次性记录当成长期知识 | 移到临时记录或归档，不进入 active SOP |
| 只写流程，不写验收标准 | 补输入、输出、通过标准和失败处理 |
| 只改正文，不改路由 | 同步补 `routing.md` 或说明为什么无需路由 |
| 只修本地，不回读远端 | push 后回读远端分支、文件树和提交元数据 |

## 4. 不应过度纠偏

纠偏是为了恢复正确方向，不是把简单任务变复杂。

- 低风险错别字不需要扩成治理变更。
- 一次性尝试失败不一定需要沉淀。
- 能用轻量回读证明的，不必做全量审计。
- 如果纠偏成本明显高于收益，说明限制并继续完成主任务。

## 5. 收口检查

完成纠偏后，至少确认：

- 当前结论是否已区分事实和推测。
- 当前入口是否与 `routing.md` 一致。
- 输出是否让读者快速看懂结论、依据、风险和下一步。
- public 内容是否仍不含凭证、运行态和敏感业务信息。
- 如果修改了框架结构，是否同步更新 README、路由或治理索引。
