# 会话与运行态留存规则

> 用于区分长期知识、协作账本、运行态报告和无价值噪音。

## 1. 基本原则

会话和运行态不是长期知识真源。

长期应沉淀的是：

- 经过整理的规则。
- 可复用 SOP。
- 稳定知识卡。
- 决策记录。
- 工具说明。
- 脱敏案例摘要。

不应长期沉淀的是：

- 原始聊天记录。
- prompt、stdout、stderr 原文。
- 原始日志、数据库结果、接口返回。
- 浏览器状态、缓存、临时导出。
- 含凭证或敏感上下文的运行输出。

如果运行态里有价值，应先提炼成规则、SOP、知识卡或报告摘要，再决定是否清理原始材料。

## 2. 内容分类

| 类型 | 例子 | 默认处理 |
|------|------|----------|
| 长期真源 | `governance/`、`sops/`、`knowledge/`、`templates/` | 进入 Git，按变更控制维护 |
| 协作账本 | reviewer 请求摘要、状态、结论、runId | 可短期保留；不存完整正文和 secret |
| 运行态状态 | 自动化 state、同步 summary、doctor 报告 | 本机保留；不进入 public |
| 临时输出 | 草稿、一次性导出、调试文件 | 过期后清理或隔离 |
| 原始证据 | 日志、截图、接口响应、数据库结果 | 默认不进 starter；必要时只保留脱敏摘要 |
| 凭证材料 | token、cookie、私钥、连接串 | 不保留、不同步、不分享 |

## 3. 留存策略

| 策略 | 适用内容 | 要求 |
|------|----------|------|
| 长期维护 | 已验证 SOP、知识卡、治理规则 | 有清晰入口和 owner |
| 短期保留 | 运行报告、协作状态、同步 summary | 有生成时间和用途 |
| 隔离观察 | 不确定是否仍有价值的旧产物 | 先移动到隔离区，保留到期日 |
| 立即排除 | 凭证、密钥、会话原文、运行态噪音 | 不进 Git，不进 public |

团队可以自定义保留周期；建议先从 30 天短期保留和 90 天归档复核开始，不要一上来做不可回退删除。

## 4. 清理前检查

执行清理或归档前，至少确认：

1. 清理对象属于哪一类。
2. 是否读取正文；只做体积盘点时默认不读正文。
3. 是否含凭证、业务资料或个人信息。
4. 是否已有对应的 SOP、知识卡、报告摘要或决策记录。
5. 删除是否可回退；不可回退时必须明确说明。
6. 是否会影响活跃自动化、会话恢复或正在运行的工具。

## 5. 不进入 public starter 的内容

- `runtime/` 下除 README 和公开模板之外的真实状态。
- 自动化真实 state、日志、summary。
- 协作原始 request / response / prompt。
- 本机 host 的会话历史、数据库、索引和缓存。
- 真实 `.env`、`.toml`、`.json` 配置。
- 截图、表格、压缩包、PDF、HAR、SQLite 等原始附件。

public starter 可以保留这些内容的治理规则和目录说明，但不保留真实数据。

## 6. 沉淀流程

当一次会话或运行报告值得复用时：

1. 提取可复用结论。
2. 删除或泛化业务上下文。
3. 判断落点：`knowledge/`、`sops/`、`governance/`、`case-studies/`。
4. 更新 `routing.md` 或索引。
5. 原始运行态保留在本机短期区域，按周期清理。

一句话原则：

```text
长期保存结论和规则，短期保存证据和状态，不保存凭证和噪音。
```
