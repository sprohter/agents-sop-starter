# 变更控制规则

> 修改框架规则、路由、skill、同步任务或共享 SOP 结构前，先用本规则控风险。

## 1. 变更分级

| 等级 | 范围 | 默认处理 |
|------|------|----------|
| L0 文案小修 | 错别字、格式、单处说明补充 | 可直接改，改后回读确认 |
| L1 框架文档 | `governance/`、`routing.md`、`contract.md`、共享模板 | 说明影响面，同步更新索引或路由，做轻量扫描 |
| L2 自动化或工具 | 同步脚本、定时任务、本地 adapter、后台任务 | 需要明确目标、日志、状态文件、停止方式，优先 dry-run |
| L3 外部写入或破坏性动作 | push、发布、删除、force update、外部系统写入 | 需要明确授权，执行后必须回读验证 |

跨多个等级的变更，按最高等级处理。

## 2. 高风险变更前说明

执行 L2 或 L3 前，至少写清：

- 目标：影响哪个仓库、分支、文件、系统或任务。
- 内容：会写入、删除、发布或替换什么。
- 影响：谁能看到，可能破坏什么。
- 回退：如何撤销或停止；无法回退时说明原因。
- 验证：用什么 diff、命令、回读或清单证明最终状态。

## 3. Dirty Workspace 规则

不要把无关本地改动混入框架治理变更。

- 优先做定点 diff，不做整仓粗暴操作。
- 不回滚用户改动，除非用户明确要求。
- 如果当前工作区很脏，优先使用临时克隆或窄范围 export 目录操作。

## 4. Public 分享门禁

推送公开内容前：

1. 使用 public export profile。
2. 执行敏感内容扫描或等价人工扫描。
3. 确认 changed paths 只包含预期 starter 资产。
4. 确认 Git author / committer 元数据不暴露个人或组织身份。
5. push 后回读远端分支、文件树和最新提交元数据。

## 5. 删除门禁

删除只在目标清楚、影响可接受时执行。

- 旧分支删除前，先确认该分支提交已经包含在目标分支中。
- 旧仓库、release、package、wiki、pages 和 public 历史清理需要单独明确授权。
- 本地运行态清理前，确认重要证据已经沉淀到报告、工单或知识库。
