# 经验沉淀规则

> 把一次性处理变成可复用 SOP 或知识卡。

## 什么时候需要沉淀

满足任一条件即可沉淀：

- 同类问题第二次出现。
- 本次处理踩坑，未来容易重复。
- 某个判断规则需要团队统一。
- 某个流程可以培训新人。
- 用户明确要求“沉淀、固化、写成 SOP、以后照这个走”。

## 沉淀到哪里

| 内容 | 位置 |
|------|------|
| 岗位职责、接管边界 | `roles/` |
| 完整可执行流程 | `sops/` |
| 稳定事实、术语、经验卡 | `knowledge/` |
| 框架维护规则 | `governance/` |
| 可复制格式 | `templates/` |

## 沉淀流程

1. 提取本次可复用结论。
2. 去掉业务资料、个人路径和敏感信息。
3. 判断放置位置。
4. 使用对应模板创建或更新文件。
5. 更新 `routing.md`。
6. 在最终答复中说明新增或更新了什么。

## 写法要求

- 先写“什么时候用”。
- 再写“怎么做”。
- 最后写“输出什么、如何验收”。
- 保留必要证据类型，但不保留敏感证据本身。
- 一条规则只维护一个真源。

## 不沉淀什么

- 只发生一次且没有复用价值的临时过程。
- 未确认的猜测。
- 真实业务数据样例。
- 密钥、链接、个人目录。
- 与岗位 SOP 无关的聊天记录。

## 沉淀成熟度

| 状态 | 说明 |
|------|------|
| `draft` | 刚提炼，允许继续改 |
| `active` | 已验证，可推荐执行 |
| `deprecated` | 已替代，不再推荐 |
