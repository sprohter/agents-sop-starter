# 框架维护规则

> 让这套基础框架长期保持可分享、可扩展、无业务污染。

## 1. 维护目标

- 可直接发给其他人。
- 可被 agent 快速读取。
- 岗位和 SOP 可以持续增长。
- 不混入个人资料、业务资料和临时噪音。

## 2. 分层规则

| 问题 | 放哪里 |
|------|--------|
| 谁负责、什么时候接管 | `roles/` |
| 事情怎么一步步做 | `sops/` |
| 稳定事实、术语、口径 | `knowledge/` |
| 框架怎么维护 | `governance/` |
| 可复制空模板 | `templates/` |
| agent 怎么接入 | `config-templates/` |

## 3. 新增内容前检查

新增文件前先问：

1. 是否已有相同主题？
2. 是否应该补到已有 SOP，而不是新建？
3. 是否需要同步更新 `routing.md`？
4. 是否包含敏感信息或业务资料？
5. 是否能被其他岗位复用？

## 4. 最小联动

新增或显著修改文件后，至少检查：

- `README.md` 是否需要补目录说明
- `routing.md` 是否需要补路由
- 相关 `roles/*.md` 是否需要补 SOP 链接
- 是否违反 `privacy-and-share-boundary.md`

## 5. 版本建议

- 小改：直接更新对应文件。
- 大改目录结构：补一条决策记录，使用 `templates/decision-record.template.md`。
- 废弃文件：保留替代入口，状态标记为 `deprecated`。

## 6. 禁止事项

- 在多个文件重复定义同一条规则。
- 把岗位卡写成完整流程文档。
- 把 SOP 写成历史流水账。
- 把真实业务资料当示例。
- 把个人本机路径写成团队标准。
