# 底座治理体检清单

> 用于定期检查 agent 底座是否开始变乱、变重、变重复或偏离分享边界。

## 1. 什么时候跑

满足任一条件即可做一次治理体检：

- 新增了一批 SOP、knowledge、skill 或工具。
- 准备对外分享或推送 public starter。
- 路由开始不准，agent 经常读错入口。
- 文档越来越多，新人不知道先看哪里。
- 自动化、工具或运行态目录开始膨胀。
- 阶段性复盘，例如每月或每季度。

## 2. 入口检查

检查：

- 是否只有一个主入口：`README.md`。
- 是否只有一个任务路由入口：`routing.md`。
- 新增岗位、SOP、工具和治理规则是否能从路由找到。
- 是否有长文档仍在承担入口职责。

建议：

- 保留短入口，长说明降级为详细参考。
- 路由不准时优先补 `routing.md` 和 `routing-maintenance.md`。

## 3. 重复与冲突检查

检查：

- 同一条规则是否写在多个文件里。
- SOP 和 knowledge 是否重复描述同一流程。
- public / private / internal 边界是否互相冲突。
- 新规则是否覆盖旧规则但没有标记迁移。

建议：

- 一条规则只保留一个真源。
- 被替代文件标记 `deprecated`，写清替代入口。
- 不确定的新经验先放 draft 或候选，不直接覆盖稳定规则。

## 4. 沉淀质量检查

检查最近新增沉淀：

- 是否有明确触发场景。
- 是否写清输入、步骤、输出和验收。
- 是否去掉了业务资料、个人路径和敏感信息。
- 是否补了路由、索引或岗位引用。
- 是否属于可复用经验，而不是一次性流水账。

建议：

- 可培训、可复用、可验收的内容进入 `sops/`。
- 稳定事实、术语、判断口径进入 `knowledge/`。
- 框架维护规则进入 `governance/`。
- 脱敏案例摘要进入 `case-studies/`。

## 5. 能力与工具检查

检查：

- 工具是否有 owner、用途、输入输出、权限、日志和停用方式。
- 默认模式是否只读或 dry-run。
- 是否有真实凭证、真实 endpoint 或运行态混入 Git。
- 是否有能力已经退役但仍在路由中。

建议：

- 工具说明放 `tools/`。
- skill 说明放 `skills/`。
- peer / mesh 接入放 `mesh/`。
- 真实配置和状态留在本机 secret / runtime 区，不进入 public。

## 6. 自动化检查

检查每个自动化是否写清：

- owner
- target
- schedule
- state
- logs
- stop method
- permissions

建议：

- 周期任务默认续接已有线程或固定状态，不默认新建会话。
- 同步任务 push 后必须回读远端确认。
- 失败 summary 不能被旧 summary 冒充。

## 7. 分享边界检查

Public 分享前确认：

- 不含凭证密钥、cookie、私钥、连接串。
- 不含真实客户、订单、项目、系统、库表、接口、内部链接。
- 不含本机绝对路径、真实主机、IP、群 ID、账号。
- 不含原始日志、截图、表格、会话正文、运行态报告。
- 新增文件已进入 allowlist，并通过敏感扫描。

如果内容有价值但含敏感细节，先抽象成通用规则，再决定是否进入 public。

## 8. 输出格式

体检输出建议固定为：

```text
本次治理体检结果

1. 主入口与路由
2. 重复与冲突
3. 沉淀质量
4. 能力与工具
5. 自动化与运行态
6. 分享边界
7. 归档候选
8. 建议动作：P1 / P2 / P3
```

优先处理 P1：

| 优先级 | 含义 |
|--------|------|
| P1 | 安全、路由、外部写入或 public 分享风险 |
| P2 | 重复、过时、入口不清 |
| P3 | 文案、格式、低影响整理 |
