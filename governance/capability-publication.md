# 能力发布与工具边界

> 用于判断脚本、skill、adapter、mesh endpoint 或其他 agent 能力能否进入 starter 或共享包。

## 1. 基本观点

脚本能力、工具能力和 agent 能力本身是有价值的资产。

可以分享的是：

- 能力设计。
- 目录结构。
- 输入输出契约。
- 权限边界。
- 公开安全脚本。
- 使用占位符的配置样例。
- 如何接入、如何验证、如何停用。

不能分享的是：

- 真实凭证、密钥、cookie、连接串。
- 真实内部主机、私有 URL、群 ID、账号、IP。
- 原始运行态、日志、缓存、会话正文、一次性输出。
- 真实业务 payload、客户数据、订单数据、事故现场。

目标不是把本机环境原样复制出去，而是让读者知道“这个能力为什么存在、怎么设计、怎么安全接入”。

## 2. 能力分级

| 类型 | 示例 | Public Starter | Private Backup |
|------|------|----------------|----------------|
| C0 凭证能力 | token、私钥、真实账号配置 | 禁止 | 禁止 |
| C1 运行态能力 | 缓存、日志、临时状态、浏览器会话 | 禁止 | 禁止 |
| C2 内部连接器 | 真实内部系统 adapter、生产环境 endpoint | 禁止 | 仅保留脱敏说明和模板 |
| C3 可复用工具 | 通用脚本、只读检查器、格式转换器 | 脱敏后允许 | 允许 |
| C4 agent 能力 | skill、SOP worker、mesh peer 说明 | 脱敏后允许 | 允许 |
| C5 能力设计 | 权限模型、注册表、验证清单、接入模式 | 允许 | 允许 |

## 3. 发布前必须写清

共享一个能力前，至少补齐：

| 字段 | 要求 |
|------|------|
| name | 能力名，稳定可引用 |
| owner | 维护角色或团队 |
| purpose | 解决什么问题 |
| inputs | 输入参数、文件或事件 |
| outputs | 输出文件、状态、报告或外部动作 |
| permissions | 允许读写什么，禁止触碰什么 |
| side effects | 是否会写文件、发消息、调用外部系统 |
| secrets | 凭证从哪里读取，文档中只能写占位符 |
| logs | 日志和状态写到哪里 |
| stop | 如何停用、回滚或禁用 |
| verification | dry-run、doctor、smoke 或回读验证方式 |

## 4. Public 分享写法

Public starter 中推荐保留：

- `README.md` 或工具卡，说明用途和边界。
- 脱敏脚本，默认 dry-run 或只读优先。
- `.example` / `.template` 配置，所有敏感项用 `<placeholder>`。
- `doctor` / `readiness` / `smoke` 检查。
- 权限、日志、停止方式和失败处理。

Public starter 中不要保留：

- 真实 `.env`、`.toml`、`.json` 配置。
- 真实 webhook、真实 peer、真实 endpoint。
- 可直接触发高风险写入的默认参数。
- 含业务数据的 sample payload。
- 一次性任务脚本，除非已抽象成通用工具。

## 5. 能力注册表示例

团队可以维护一个轻量能力目录：

| Capability | Type | Visibility | Default Mode | Risk | Entry | Notes |
|------------|------|------------|--------------|------|-------|-------|
| `<name>` | `<skill/tool/adapter/mesh>` | `<public/internal/private>` | `<readonly/dry-run/write>` | `<L0-L5>` | `<path>` | `<short-note>` |

注册表只记录能力边界，不记录 secret 值和运行态值。

## 6. 高风险能力门禁

命中以下任一条件，至少按 L4 治理复核处理；如果会真实外部写入，按 L5 处理：

- 会向外部系统写入、发布、删除或发送通知。
- 会修改权限、凭证、计划任务或同步目标。
- 会影响团队共享仓库、公开分享仓库或生产环境。
- 会读取或处理敏感数据。
- 会暴露本机能力给其他 agent 或 peer。

高风险能力必须具备：

1. 默认 dry-run 或只读模式。
2. 明确的人工确认点。
3. 执行后回读验证。
4. 失败摘要和可停止方式。
5. 不把凭证和运行态写入 Git。

## 7. 维护要求

- 新增 public 能力时，同步更新 `tools/README.md`、`skills/README.md`、`mesh/README.md` 或对应索引。
- 新增触发入口时，同步更新 `routing.md`。
- 能力退役时，保留替代入口或迁移说明。
- 发现能力越权或边界不清时，先下线默认写入路径，再补治理规则。
