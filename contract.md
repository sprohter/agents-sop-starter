# Agent 最小公约

> 所有接入本框架的 agent 默认遵守本文件。

## 1. 先读入口

执行任务前默认先读：

1. `README.md`
2. `contract.md`
3. `routing.md`

再按 `routing.md` 命中的路径读取岗位卡、SOP 或知识卡。

## 2. 输出规则

- 默认先给结论，再给依据、风险和下一步。
- 内容要可扫读，避免连续大段文字。
- 对比类优先表格，流程类优先步骤。
- 复杂内容分为“主结论”和“补充信息”。
- 不确定内容必须标注“推测 / 待确认”，不能写成事实。

## 3. 证据优先

任何结论都应尽量指向可复核依据：

- 文件路径
- 记录编号
- 日志片段
- 截图说明
- 配置项
- 数据来源
- 会议或需求来源

找不到依据时，直接说明“未找到依据”。

## 4. 安全边界

默认禁止写入本框架：

- 账号、密码、token、cookie、证书
- 含密钥的连接串
- 客户、订单、合同、人员隐私等敏感资料
- 个人本机绝对路径
- 未脱敏的业务数据样例

如果必须说明配置方式，只写占位符，例如 `<your-token>`、`<workspace-root>`、`<issue-system-url>`。

## 5. 高风险动作确认

以下动作必须先说明影响并等待确认：

- 删除文件或数据
- 修改生产配置或共享配置
- 向外部系统发送消息、公告、工单、审批
- 批量修改状态
- 启动常驻后台任务、计划任务或开机启动项

一次确认只对当次动作有效。

## 6. 本机后台任务规则

新增或修改后台任务、定时任务、watchdog、bridge/server、常驻脚本时，默认不得打断用户桌面工作。

- Windows 下优先使用隐藏窗口包装、服务模式或等价无窗口启动方式。
- 不直接以前台窗口启动 `cmd.exe`、`powershell.exe`、`node.exe`、`python.exe` 等后台入口。
- stdout / stderr 应写日志或状态文件。
- 失败应可追踪，不弹出干扰窗口。

## 7. 文件放置规则

| 内容 | 默认位置 |
|------|----------|
| 岗位职责、触发词、常用 SOP 列表 | `roles/` |
| 可重复执行的标准流程 | `sops/` |
| 稳定经验、口径、术语、事实 | `knowledge/` |
| 框架维护、沉淀、隐私、路由规则 | `governance/` |
| 可复制模板 | `templates/` |
| 本地 agent 接入样板 | `config-templates/` |

## 8. 交付自检

最终答复前检查：

- 是否回答了最新请求
- 是否有明确结论
- 是否说明产物路径
- 是否说明未完成项或风险
- 是否避免写入敏感信息
- 是否把可长期复用内容沉淀到正确目录
