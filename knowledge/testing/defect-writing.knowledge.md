# 缺陷写作知识卡

status: active
owner: qa-team
last_updated: 2026-06-05

## 结论

缺陷描述默认遵循“标题短、正文最小可复现、证据分层”的原则。不要把缺陷写成调查报告。

## 标题规则

- 直接写功能点 + 异常现象
- 避免只写“有问题”“异常”“Bug”
- 标题让接手人不看正文也能知道大概问题对象

## 正文最小结构

```text
【步骤】
最少必要的复现入口、样本对象和触发动作

【结果】
实际异常现象、报错或关键差异

【期望】
一句话说明正确行为
```

## 写作原则

- 步骤只写复现必须信息
- 结果优先保留原始异常或关键差异
- 期望只写正确口径，不写完整修复方案
- 日志、SQL、接口返回、截图说明等证据放补充区，不抢主干

## 什么时候要补更多证据

| 场景 | 需要补的内容 |
|------|--------------|
| 数据问题 | 样本对象、关键字段或最小数据差异 |
| 接口问题 | 错误码、关键返回字段或调用条件 |
| 日志问题 | 关键时间、关键报错或定位标识 |
| 页面问题 | 截图里看不出的账号、对象、步骤或差异 |

## 不建议的写法

- 先写一大段背景，再到最后才说异常
- 把完整排查过程塞进 `【结果】`
- 缺少复现对象，导致接手人还要追问“用哪条数据复现”
